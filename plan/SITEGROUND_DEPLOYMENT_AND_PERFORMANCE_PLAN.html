Recommended source optimizations before production

These are separate approval-gated implementation batches. None is required merely to document or configure SiteGround.

1. Right-size homepage service icons

Generate a non-cropped 150–160px attachment size and let core Image blocks emit srcset and sizes. Preserve attachment IDs, links, alt text, editor geometry, and visible 52px rendering. Remove inappropriate high priority from service icons that begin roughly 2,000px below the mobile viewport.

2. Consolidate LMHG Site Core maintenance gates

Replace roughly 28 public-request migration, seed, cleanup, and synchronization checks with one lightweight Site Core data-version gate. Run existing migrations in their current order only when the version is behind, while retaining all current markers and audit reports.

3. Bound two content queries

Batch the FAQ-index query instead of issuing one query per FAQ set. Apply the three-card limit inside the Related Articles SQL query rather than loading every match and slicing in PHP. Preserve output ordering and content exactly.

4. Remove small request-path redundancies

Do not enqueue review CSS on a homepage with no review showcase. Load the administration-only module only for administration requests. Add cheap no-op guards and request-local dimension memoization to the accessibility HTML buffer without removing any accessibility correction.

5. Limit CSS cleanup to proven legacy rules

Remove only selectors proven unused across tracked template parts, database template-part state, and every rendered page archetype. Keep the exact style.css and theme.json.styles.css mirror. Do not introduce a new CSS architecture for a stylesheet that is currently about 10.6KB compressed.

MariaDB and WordPress database plan

The current development database is approximately 10.4MB, with about 84KB of normally autoloaded options, zero orphaned post metadata, and a development InnoDB read-hit ratio above 99.99%. The database does not justify manual tuning or schema surgery.

Do

  • Export a consistent, checksummed MariaDB backup.
  • Verify SiteGround's target engine and version before import.
  • Use a portable WordPress SQL export without environment-specific paths or secrets.
  • Run immutable-content and critical-option comparisons after import.
  • Retain rollback data until production acceptance is complete.

Do not

  • Do not add custom indexes without EXPLAIN or slow-query evidence.
  • Do not run broad OPTIMIZE TABLE operations as routine deployment work.
  • Do not delete revisions, scheduler rows, or transients manually.
  • Do not copy Docker memory settings into SiteGround's managed database environment.
  • Do not present post-import runtime rows as immutable source parity.

SiteGround identifies MariaDB 10.11 and 10.6 as recommended WordPress-compatible database versions. Codex must still confirm the actual managed target before importing. See SiteGround's WordPress requirements.

Scheduled maintenance

Codex should replace traffic-driven WP-Cron with a SiteGround real cron only after proving the scheduled command works. A reasonable starting point is twice hourly:

cd /home/customer/www/louisvillementalhealth.org/public_html
wp cron event run --due-now >/dev/null 2>&1

The path must be resolved from the actual SiteGround account. Disable request-driven WP-Cron only after a manual invocation succeeds and due events drain without duplicate execution. See SiteGround's real-cron procedure.

Codex-owned deployment sequence

  1. Release authority: verify explicit approval, clean main, local/remote equality, required review-branch merges, and the exact production commit.
  2. Inventory: record WordPress, PHP, target database, active/inactive plugin versions, theme version, canonical URL, SiteGround document root, cache state, and current DNS/TLS behavior.
  3. Rollback set: create and verify a database backup, wp-content archive, current production file inventory, and SiteGround restore point before mutation.
  4. Staged transfer: deploy tracked theme/plugin files without secrets or licensed binaries; preserve uploads and required installed plugins.
  5. Database import: import once, perform serialized URL replacement with WordPress-aware tooling if needed, and compare critical content/options with the source snapshot.
  6. Canonical configuration: set home/siteurl, HTTPS, permalinks, noindex/indexability policy, Rank Math sitemap behavior, and production-safe environment constants.
  7. Application validation: verify administration, Gutenberg editors, REST, redirects, sitemap/schema, images, CTA destinations, accessibility, mobile layouts, and application logs before enabling aggressive optimization.
  8. SiteGround optimization: enable one control at a time, purge and warm caches, then collect raw headers, rendered parity, and Lighthouse evidence after each material control.
  9. Cutover: change public routing only after all blocking checks pass. Retain the rollback set and monitor public errors, cache behavior, scheduled actions, and form/CTA destinations.
  10. Closeout: publish the production commit, configuration evidence, validation results, known deferrals, and rollback window. Never include credentials in the report.

Required repository checks

  • PHP syntax checks for every changed LMHG Site Core file and git diff --check.
  • npm run test:gutenberg:self, then live all-editable Gutenberg verification with safe ephemeral-administrator cleanup.
  • npm run test:page-data, npm run test:responsive-h1, and npm run test:responsive-text.
  • npm run test:site-core:accessibility, npm run test:site-core:redirects, and Article runtime verification where applicable.
  • Exact style.css versus theme.json.styles.css comparison after every CSS change.
  • Raw and rendered production checks after every SiteGround purge/warm cycle; one stale rendering is not sufficient evidence of failure or success.

Deferred AI-crawler and edge/WAF task

Post-deployment only: SiteGround has previously returned connection-level empty responses for selected AI crawler user agents even when robots.txt allowed them and static text files were requested. This is separate from WordPress performance work and must not block the initial optimization implementation.

After production stabilizes, Codex will:

  1. Verify normal-browser, search-crawler, and named AI-crawler access to /robots.txt, /llms.txt, the sitemap, and one static asset.
  2. Compare status, headers, TLS behavior, and response bodies using exact user-agent evidence.
  3. Inspect WordPress, Speed Optimizer, Security Optimizer, .htaccess, and SiteGround controls without assuming the block originates in WordPress.
  4. If connection-level drops persist, provide SiteGround support with timestamped curl evidence, source IP, DNS target, affected user agents, and successful control requests.
  5. Reverify crawler access after any SiteGround edge/WAF adjustment without weakening general security controls.

Decision summary

  • Use SiteGround's native caching stack deliberately, with Speed Optimizer as the single integration surface.
  • Prioritize right-sized images and request-path cleanup over theme rearchitecture.
  • Use Memcached for persistent WordPress object caching, but make no speculative MariaDB schema or server changes.
  • Keep mobile responsiveness and Gutenberg validity as hard release gates.
  • Codex owns deployment execution and evidence collection after explicit production approval.
  • Handle AI-crawler edge blocking as a separate post-deployment SiteGround task.
